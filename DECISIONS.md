## 判断ログ一覧

### [ADR-001] 永続化に SQLite(JDBC) を採用しモックDBを置換

**日時**：2026-06-27

**状況**：
スターターの `Database` は `findByEmail` が常に null、`save` がタイムスタンプID という壊れたモックで、重複チェックが実質機能していなかった。テスト時に差し替えられる実DBが必要。

**決定**：
SQLite + JDBC で実装。ローカル/テストはインメモリ（`jdbc:sqlite::memory:`）、本番はファイルDBを想定。依存は Maven(`pom.xml`)で管理。

**理由**：
追加のDBサーバ不要で導入が軽く、インメモリでテストが完結する。ハッカソンの規模に見合う。

**トレードオフ**：
本格的なRDB（PostgreSQL等）は使わない。スケール・並行性は割り切る。

### [ADR-002] Repository層を廃し Active Record（MVC）に簡素化

**日時**：2026-06-27

**状況**：
当初は `UserRepository` インターフェース + `SqliteUserRepository` で永続化を抽象化していた（ADR-001 の実装）。しかし小さな登録機能には層が多く冗長だと判断した。

**決定**：
Repository/Service 層をやめ、Model(`User`) が DBアクセスを直接持つ Active Record とし、`model` / `view` / `controller` の MVC 3層に集約した。

**理由**：
ドメインが小さく、永続化を差し替える余地より見通しの良さを優先。MVC で各クラスの役割が明快になる。

**トレードオフ**：
永続化の差し替え（別DBやモック）はしにくくなる。DB接続を `static` で持つグローバル状態を許容。差し替え不能になるテスト容易性はインメモリSQLite（ADR-004）で担保する。

### [ADR-003] register() で authProvider を分岐し、OAuth はパスワード検証をスキップ

**日時**：2026-06-27

**状況**：
従来のパスワード登録に加え、OAuth 登録を扱いたい。OAuth の authorization code grant は完了済みの前提とする。

**決定**：
`RegisterInput` に `authProvider`（PASSWORD / GOOGLE / GITHUB）を持たせ、Controller で分岐。PASSWORD のみパスワード検証とハッシュ化を行い、OAuth は `password_hash` を null で保存する。メール/名前の共通バリデーションは全方式で共有。

**理由**：
認証方式ごとの差分を `register()` 1か所に閉じ込め、共通処理と分離できる。

**トレードオフ**：
provider ごとにクラスを分ける（Strategy パターン）はしない。種類が増えると分岐が膨らむが、現状の2系統では分岐で十分シンプル。

### [ADR-004] テストはインメモリSQLiteに対して実施

**日時**：2026-06-27

**状況**：
永続化を含む登録フローを検証したい。

**決定**：
JUnit で各テストごとに `jdbc:sqlite::memory:` へ接続し直し、まっさらな状態で検証する。

**理由**：
モックではなく実SQLのスキーマ・UNIQUE制約・分岐まで一気通貫で確認できる。接続コストが低くテスト間の独立性も保てる。

**トレードオフ**：
メール送信の抽象を外した（ADR-002）ため、送信内容の検証はしない（標準出力のみ）。
